Custom Home Assistant integration for receiving real-time GPS location data from mobile devices and vehicle trackers via webhooks.

---

## When to use this integration

### ✅ Good use cases

- **Backup location source** — if the official Home Assistant Companion app fails or is unavailable, any OsmAnd-compatible app or Traccar Client can keep sending location to HA independently.
- **Location without installing the HA app** — the Companion app grants the sender full access to your Home Assistant instance. Using a dedicated tracking app (OsmAnd, Traccar Client) with this integration exposes only the webhook endpoint, with no HA credentials involved.
- **Reuse existing apps** — if you already use [Traccar Client](https://www.traccar.org/client/) or [CarStatsViewer](https://github.com/Ixam97/CarStatsViewer), this integration receives their data directly without requiring a separate [Traccar Server](https://www.traccar.org/server/) installation.

### ❌ When this is not enough — consider Traccar Server instead

- **Historical data / offline buffering** — this integration only processes real-time pushes. If a device goes offline and resumes later, any data sent while offline is lost. Traccar Server buffers and stores all positions.
- **Route tracking and trip history** — Traccar Server provides a full map-based route viewer with trip replays. This integration has no persistent history beyond what HA's own recorder captures.

---

## Requirements

- **Home Assistant ≥ 2026.3.0** — required for local brand icons (`brand/icon.png`) support.
- **HACS ≥ 2.0.0** — required for installation via HACS.
- A Home Assistant instance **accessible from the internet** so devices can reach the webhook URL. See the [HA remote access documentation](https://www.home-assistant.io/docs/configuration/remote/) for setup options (Nabu Casa, reverse proxy, etc.).
- HTTPS is strongly recommended when the webhook URL is exposed to the internet — see the [Security Note](#security-note) section.

## Supported Protocols

| Protocol | Method | Reference |
|----------|--------|-----------|
| OsmAnd / Traccar — query parameters | `GET` or `POST` (form) | [OsmAnd protocol](https://www.traccar.org/osmand/) |
| OsmAnd / Traccar — JSON body | `POST application/json` | [OsmAnd JSON](https://www.traccar.org/osmand/) (Traccar Client ≥ 9.0) |
| CarStatsViewer | `POST application/json` | [CarStatsViewer API](https://github.com/Ixam97/CarStatsViewer/blob/master/docs/APIDOC.md) |

The OsmAnd webhook **auto-detects the format** on every request. The same URL handles both the classic query-parameter format and the newer JSON body format.

## Configuration

Go to **Settings → Devices & Services → Add Integration** and search for **Location Receiver**.

### Step 1 — Device Name & Type

| Field | Description |
|-------|-------------|
| Device Name | Label used for the HA device and its entities |
| Device Type | `OsmAnd / Traccar` or `CarStatsViewer (CSV)` |

### Step 2 — OsmAnd Webhook Mode *(OsmAnd only)*

| Mode | Description |
|------|-------------|
| **Individual webhook** | Dedicated URL per device. Recommended for most users. |
| **Global webhook** | One shared URL for all OsmAnd devices, identified by `device_id`. Configured once for the entire integration. |

When using **Global webhook** you must enter the **Device ID** that the tracker sends (OsmAnd: *Plugins → Trip Recording → Online Tracking → Device identifier*).

### Step 3 — Webhook ID

A cryptographically secure random ID is generated by default. You may define a custom ID if required — see [Security note](#security-note).

After setup, the webhook URL appears in the integration entry card:

```
http://your-ha-instance:8123/api/webhook/<webhook_id>
```

### Reconfiguration

Open the integration entry → **Configure** to view or change the webhook ID, OsmAnd mode, or bound device ID. The current webhook URL is always shown at the top.

---

## Security Note

> The webhook ID is the **only secret** protecting your endpoint.
>
> - **Auto-generated IDs**: 64 hex characters (256 bits of entropy) — essentially unguessable.
> - **Custom IDs**: convenient but must be kept secret. Short or predictable values significantly reduce security.
>
> Use HTTPS in production. If Home Assistant is internet-facing, place it behind a reverse proxy with TLS.

## OsmAnd / Traccar Setup

### Format 1 — Query parameters (classic OsmAnd)

In OsmAnd go to **Plugins → Trip Recording → Settings → Online Tracking**:

| Field | Value |
|-------|-------|
| Web address | `http://your-ha:8123/api/webhook/<webhook_id>` |
| Tracking interval | As desired (e.g. 5 s) |
| Buffer size | 1 |

Supported parameters:

| Parameter | Alt. name | Description |
|-----------|-----------|-------------|
| `id` | `deviceid` | Device identifier (mandatory) |
| `lat` | | Latitude |
| `lon` | | Longitude |
| `timestamp` | | Unix seconds/ms or ISO 8601 |
| `speed` | | Speed in m/s → converted to km/h |
| `bearing` | `heading` | Heading in degrees |
| `altitude` | | Altitude in meters |
| `accuracy` | | GPS accuracy in meters |
| `batt` | | Battery level (%) |
| `charge` | | Charging state (`true`/`false`) |

### Format 2 — JSON body (Traccar Client ≥ 9.0)

Set the Traccar Client server URL to the webhook URL. The client sends `POST application/json`:

```json
{
  "device_id": "my-tracker",
  "location": {
    "timestamp": "2024-01-01T00:00:00.000Z",
    "coords": {
      "latitude": -23.5505,
      "longitude": -46.6333,
      "accuracy": 10,
      "speed": 0,
      "heading": 0,
      "altitude": 760
    },
    "battery": {
      "level": 0.85,
      "is_charging": false
    },
    "activity": { "type": "in_vehicle" },
    "is_moving": true,
    "odometer": 0,
    "event": "motionchange"
  }
}
```

> `battery.level` is a decimal fraction `0.0–1.0`; converted to % automatically.
> `speed` is m/s; converted to km/h automatically.

### Global Webhook — Unknown Device Notification

When an unrecognised `device_id` arrives on the global webhook:

1. A **persistent notification** appears in HA with the unknown ID and setup instructions.
2. A warning is written to the HA log.
3. After **2 distinct unknown IDs** have triggered notifications, further unknowns are silently dropped (DoS protection).

## Testing with curl

Use these examples to verify that your webhook is reachable and that the integration is processing data correctly. Replace `your-ha:8123` and `<webhook_id>` with your actual values.

A successful request returns HTTP `200 OK` with an empty body. After sending, check **Settings → System → Logs** for `Location Receiver` entries and verify that the entities update in **Settings → Devices & Services**.

---

### OsmAnd — GET (query parameters)

Minimum required: device identifier + latitude + longitude (or battery level).

```bash
curl -v "http://your-ha:8123/api/webhook/<webhook_id>?id=my-tracker&lat=-23.5505&lon=-46.6333&batt=85"
```

With more fields:

```bash
curl -v "http://your-ha:8123/api/webhook/<webhook_id>?\
id=my-tracker\
&lat=-23.5505\
&lon=-46.6333\
&timestamp=1700000000\
&speed=13.8\
&bearing=270\
&altitude=760\
&accuracy=8\
&batt=85\
&charge=false"
```

---

### OsmAnd — POST JSON (Traccar Client ≥ 9.0)

Minimum required: `device_id` + `location` object with `coords` (lat/lon) or `battery` level.

```bash
curl -v -X POST "http://your-ha:8123/api/webhook/<webhook_id>" \
  -H "Content-Type: application/json" \
  -d '{
    "device_id": "my-tracker",
    "location": {
      "timestamp": "2024-01-01T12:00:00.000Z",
      "coords": {
        "latitude": -23.5505,
        "longitude": -46.6333,
        "accuracy": 8,
        "speed": 13.8,
        "heading": 270,
        "altitude": 760
      },
      "battery": {
        "level": 0.85,
        "is_charging": false
      },
      "activity": { "type": "in_vehicle" },
      "is_moving": true
    }
  }'
```

Battery-only (no GPS position) — also accepted:

```bash
curl -v -X POST "http://your-ha:8123/api/webhook/<webhook_id>" \
  -H "Content-Type: application/json" \
  -d '{
    "device_id": "my-tracker",
    "location": {
      "timestamp": "2024-01-01T12:00:00.000Z",
      "coords": {},
      "battery": {
        "level": 0.72,
        "is_charging": true
      }
    }
  }'
```

### Expected rejection examples

These requests will be **rejected** (HTTP 200 returned but data discarded, warning logged):

```bash
# Missing device identifier — OsmAnd GET
curl "http://your-ha:8123/api/webhook/<webhook_id>?lat=-23.5505&lon=-46.6333"

# Neither lat/lon nor battery — OsmAnd GET
curl "http://your-ha:8123/api/webhook/<webhook_id>?id=my-tracker&altitude=760"

# Neither lat/lon nor battery_level — CarStatsViewer POST
curl -X POST "http://your-ha:8123/api/webhook/<webhook_id>" \
  -H "Content-Type: application/json" \
  -d '{"vehicle_id": "my-car", "heading": 90}'
```

## Troubleshooting

- Check **Settings → System → Logs** for entries prefixed with `ha_location_receiver`
- Verify the webhook URL is reachable from the device (`curl -v <url>`)
- For OsmAnd query-parameter format, ensure there is **no trailing slash** on the URL
- Entities show **Unavailable** until the first webhook request is received
- Use the `osmand_format` attribute to confirm which format is being parsed
- For Traccar Client JSON format, ensure version ≥ 9.0
